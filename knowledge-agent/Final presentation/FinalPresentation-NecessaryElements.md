# Final Presentation Necessary Elements

Purpose: Ensure we cover every required grading element in a 20-minute final presentation with a live demo.

---

## 1) Non-Negotiables (Presentation Day)

- [ ] 20-minute run-of-show is rehearsed and timed
- [ ] Live in-person demo works in presentation environment
- [ ] Backup recording exists (if live demo fails unexpectedly)
- [ ] Presentation is a continuation from midterm, not a repeat
- [ ] All team members speak and can explain their contribution
- [ ] Q&A prep completed (trade-offs, limitations, next steps)

---

## 2) Deck Structure We Must Include

- [ ] Opening: "What changed since midterm" in 60-90 seconds
- [ ] Updated system diagram (explicitly evolved from midterm)
- [ ] Refined problem statement + evidence
- [ ] Falsification results with executed stages
- [ ] Customer loop: engage -> learn -> change -> re-engage
- [ ] Demo integrated throughout (not only final 3 minutes)
- [ ] Technical process story: PRD -> MVP -> roadmap -> implementation
- [ ] Logging/debugging + test-log-fix evidence
- [ ] Success/failure criteria and where we currently stand
- [ ] Honest reflection: surprises, mistakes, what we would do differently
- [ ] Closing: impact + next falsification gate + ask for questions

---

## 3) Jason Domain Coverage (Product + System Design)

- [ ] Show midterm vs final system diagram differences
- [ ] Name what we misunderstood at midterm

- [ ] Show refined problem statement (more specific than midterm)
- [ ] Show executed falsification tests and conclusions

- [ ] Show how customer insights changed priorities/features

- [ ] Show follow-up validation after changes
- [ ] Make the feedback loop explicit and visual

## 5) Falsification Section (Use This Exact Narrative)

Slide title suggestion: **Falsification Stages: What We Tested and Learned**

- [ ] **Pre** - Pitched the idea to clients - Done
- [ ] **During** - Does it work well with the current system (does it break anything) - It works so done
- [ ] **Post** - Now that we have it built and working, the next falsification "gate" is to get real clients using it. This is where the proof is really in the pudding
- [ ] **Future** - Do clients use the documents created by this agent - Double proven (will take time for this to happen)

Presenter notes:
- [ ] Explicitly say which stages are validated now vs pending
- [ ] Avoid overstating "proven" on post/future; frame as active validation gates
- [ ] Tie each stage to observable evidence we can show
